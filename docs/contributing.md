Thank you for your interest in contributing to CaveGenLabs!

Whether you're implementing a new cave generation algorithm, improving the existing architecture, fixing bugs, or enhancing the documentation, contributions of any size are welcome.

---

## Ways to Contribute

There are many ways to help improve CaveGenLabs, including:

- Implementing new procedural cave generation algorithms
- Improving existing algorithm implementations
- Fixing bugs and improving stability
- Expanding the documentation
- Improving the user interface and visualization tools
- Suggesting new features or architectural improvements

---

## Development Philosophy

CaveGenLabs is designed around a modular architecture where each algorithm is implemented as an independent package. New algorithms should integrate with the existing configuration, parameter, and visualization systems rather than introducing custom workflows.

Whenever possible, contributions should follow the existing project structure, keep algorithms self-contained, reuse shared domain and application components, and maintain readability and consistency throughout the codebase.

---

## Adding a New Algorithm

One of the primary goals of CaveGenLabs is to make experimenting with new procedural generation techniques straightforward.

The Documentation section includes a step-by-step guide covering the entire process, from creating a new algorithm package and configuration class to implementing the generator, registering the algorithm, creating parameter inputs, and integrating the algorithm into the user interface.

See the [Documentation](documentation.md) section for the complete walkthrough.

---

## Reporting Issues

If you discover a bug or have an idea for improving CaveGenLabs, please open an issue on GitHub.

When reporting bugs, include:

1. A description of the problem
2. Steps to reproduce the issue
3. Expected behavior
4. Actual behavior
5. Screenshots, if applicable

---

## Pull Requests

Before opening a pull request, please ensure that:

- Your code follows the existing project style
- Existing functionality continues to work as expected
- Documentation is updated when appropriate
- Changes remain focused on a single feature or improvement
